Enforcing Markdown
Placement in Claude Code

Four complementary layers — from natural language intent to automated enforcement — that keep every .md file in docs/.

📄 CLAUDE.md ⚡ /create-doc Skill 🔗 PostToolUse Hook 🔌 Exit Codes 0 / 1 / 2
⚠ Context

The Problem

Why enforcement is needed at all

When Claude writes files, it doesn't always put them in the right place. A markdown file might land at the project root, inside a source directory, or wherever Claude decides is convenient in the moment.

Without enforcement, conventions only hold as long as Claude — and every team member — remembers them in every session.

✕ Without enforcement
  • • Files land wherever Claude decides
  • • Violations are silent — no error, no warning
  • • Convention relies on memory
  • • Inconsistent repo structure over time
✓ With four layers
  • • Intent set at session start via CLAUDE.md
  • • Correct placement baked into the skill
  • • Hook auto-moves misplaced files
  • • Conflicts blocked until resolved
Goal

All project markdown must live in docs/. Any .md file written outside it is a violation — unless it's on the exception list.

Layer 1

CLAUDE.md Instruction

Natural language rule loaded at every session start

CLAUDE.md is a file Claude Code reads automatically when a session starts. Any rule written in it becomes part of Claude's working context — as if you told it the rule at the beginning of every conversation.

The Rule

# PromptLab

All project markdown files go in `docs/`. Never create `.md` files
at the root or in source directories. Exception: `CLAUDE.md`.

How It Works

Claude reads CLAUDE.md on startup. The rule shapes its intent throughout the session — it behaves as if the constraint was part of its system prompt. No tooling, no scripts, no build steps.

✓ Pros
  • Zero infrastructure
    No scripts, no config, no build step — just a markdown file
  • Instant to change
    Edit one file; rule takes effect next session
  • Human-readable
    Non-technical team members can read and update it
  • Broad scope
    Applies to everything Claude does, not just specific tools
✗ Cons
  • Memory-dependent
    Claude can forget or deprioritise the rule mid-session
  • No enforcement
    Violations produce no error — they silently succeed
  • Implicit only
    Shapes Claude's intent; doesn't validate outcomes
  • Conflict risk
    Multiple CLAUDE.md files in subdirectories can conflict
Best for setting intent and default behaviour. Works well for rules that are easy to follow and rarely violated in practice.
Layer 2

create-doc Skill

A slash command that always writes to the right place

A project skill is a reusable unit of Claude behaviour defined in a single SKILL.md file inside .claude/skills/. It registers as a slash command in Claude Code's command picker.

File Structure

.claude/
└── skills/
    └── create-doc/
        └── SKILL.md     ← defines /create-doc

How to Invoke

/create-doc API reference for all endpoints
/create-doc architecture overview of the provider system
/create-doc getting started guide for new contributors

SKILL.md Frontmatter

---
name: create-doc
description: Create a markdown document in the docs/ folder.
argument-hint: "what to document (e.g. 'API reference', 'architecture overview')"
---

The description field also enables auto-triggering — Claude invokes the skill automatically when it detects phrases like "document X" or "write a guide for X" without an explicit slash command.

Steps Claude Follows

  1. 1Derive a kebab-case filename from the argument
  2. 2Check if docs/<filename>.md already exists — update in place rather than duplicate
  3. 3Read relevant source files from promptlab/ if the subject is code-related
  4. 4Write the finished document to docs/<filename>.md
✓ Pros
  • Correct placement by design
    docs/ is baked in — not a suggestion
  • Consistent output format
    Every invocation follows the same steps
  • Discoverable
    Appears in the / command picker with argument hint
  • Auto-trigger capable
    Invoked from natural language, not just slash commands
  • Version-controlled
    Skill file is tracked in git alongside the project
✗ Cons
  • Opt-in only
    Claude can still write markdown directly, bypassing the skill
  • Session reload required
    Modified skills need a Claude Code restart to register
  • Narrow scope
    Only helps when creating docs; won't catch other markdown writes
  • No fallback enforcement
    If Claude bypasses the skill, there's no correction
Best for standardising documentation creation when the team explicitly invokes the skill. Pairs with a hook for full coverage.
Layer 3

PostToolUse Hook

Automated enforcement on every Write call

A hook is a shell command registered in .claude/settings.json that Claude Code runs automatically at lifecycle events. PostToolUse fires after every tool call — including every Write — regardless of how or why Claude wrote the file.

Registration in settings.json

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "powershell -NonInteractive -NoProfile -File .claude/hooks/check-md-placement.ps1"
          }
        ]
      }
    ]
  }
}

Stdin JSON the Hook Receives

{
  "tool_name": "Write",
  "tool_input": {
    "file_path": "notes.md",   ← this is what we inspect
    "content": "..."
  }
}

Exception List

# Named files — allowed outside docs/ regardless of location
$allowedNames = @('CLAUDE.md')

# Directory subtrees — never touched
$allowedPrefixes = @(
    'docs/',          # already correct
    '.claude/',        # Claude Code internals (skills, hooks, settings)
    '.venv/',          # Python virtual environment
    'node_modules/'   # JS dependencies
)

✓ Pros
  • Always fires
    Catches every Write call, not just skill invocations
  • Self-healing
    Moves the file automatically when there's no conflict
  • No memory reliance
    Rule is in code, not in conversation context
  • Composable
    Multiple hooks can be chained for the same event
  • Version-controlled
    Script lives in .claude/hooks/ tracked in git
✗ Cons
  • Reactive, not preventive
    File is already written in the wrong place before the hook runs
  • Adds latency
    Every Write call now spawns a PowerShell process
  • Windows-specific
    .ps1 is not portable to Linux/macOS without a bash equivalent
  • No undo on exit 2
    When blocked, the misplaced file stays on disk — Claude must clean it up
Best as a safety net — catches any markdown write that slips through intent-level controls, regardless of how the file was created.
Layer 4

Hook Exit Codes

Three exit codes control what Claude Code does next

The exit code a hook returns is the primary signal back to Claude Code. It determines whether Claude continues normally, receives an informational message, or is blocked until the problem is resolved.

0
Allow
Silent pass-through

No output, no interruption. Claude Code continues exactly as it was. Use this whenever the hook has nothing to act on.

# File is in docs/ or on exception list exit 0
Claude sees outputNo
Claude continuesYes
Use whenNothing to report
In this project: file is in docs/, is CLAUDE.md, or is in .claude/, .venv/
1
Inform
Non-blocking message

Hook's stdout is shown to Claude as an informational message. Claude reads it, updates its understanding, and continues without interruption.

Write-Output "HOOK: Moved 'notes.md' → 'docs/notes.md'" exit 1
Claude sees outputYes (info)
Claude continuesYes
Use whenAction taken, Claude should know
In this project: a misplaced .md was successfully moved to docs/
2
Block
Hard stop — must resolve

Current operation is stopped. Claude reads the error and must actively resolve it — rename, merge, or fix — before proceeding.

Write-Output "HOOK BLOCKED: 'notes.md' cannot be moved 'docs/notes.md' exists." exit 2
Claude sees outputYes (error)
Claude continuesNo — must fix
Use whenIntervention required
In this project: naming conflict in docs/, or filesystem move failed

Scenario Table

Scenario File written Exit Outcome
Claude writes to correct location docs/guide.md 0 Silent pass
Claude writes CLAUDE.md CLAUDE.md 0 Named exception
Claude writes a skill definition .claude/skills/x/SKILL.md 0 .claude/ prefix exempt
Misplaced .md, no conflict in docs/ notes.md → moved to docs/notes.md 1 Auto-moved, Claude informed
Misplaced .md, name already in docs/ guide.md (conflict) 2 Blocked — Claude must rename or merge
Move fails (permissions, locked file) report.md 2 Blocked — filesystem error
✓ Pros
  • Granular control
    Three distinct severity levels in one integer
  • Exit 1 enables self-healing
    Hook can fix a problem and inform Claude without blocking
  • Exit 2 forces resolution
    Broken state cannot silently continue
  • Universal interface
    Any language, any script — exit codes are standard
✗ Cons
  • PostToolUse cannot prevent
    Exit 2 stops Claude's next action; the wrong write already happened
  • Non-obvious semantics
    1 means "error" in Unix convention; here it means "non-blocking info"
  • No structured payload
    Output is plain text — no machine-readable error type or metadata
  • Silent exit 0 hides bugs
    A crashing hook that defaults to 0 will never be noticed

No single layer is sufficient on its own. Together they form a defence-in-depth strategy where each layer catches what the previous one misses:

Without any one layer
  • ❌ No CLAUDE.md → Claude has no baseline intent
  • ❌ No skill → documentation placement is ad-hoc
  • ❌ No hook → violations that slip through are permanent
  • ❌ No exit codes → hook cannot communicate outcome
With all four layers
  • ✓ Intent set → Claude defaults to the right behaviour
  • ✓ Skill → documentation has a reliable, consistent path
  • ✓ Hook → any escape from the pattern is caught and repaired
  • ✓ Exit codes → Claude knows what happened and can react
Summary

Key Takeaways

Five things to remember from this walkthrough

1
CLAUDE.md is the foundation

Always write your rules here first. It's free, fast, and shapes Claude's default behaviour for the entire session without any tooling.

2
Skills standardise how, not just what

Use skills when a repeatable workflow needs to be enforced by convention — they make the right way the easy way.

3
Hooks are the safety net

They enforce rules at the filesystem level regardless of how Claude or the user triggered a write. The skill can be bypassed; the hook cannot.

4
Match exit code to impact

Exit 0 = nothing to say. Exit 1 = fixed it, FYI. Exit 2 = broken, you must act. Don't use 2 when 1 is enough — unnecessary blocks slow Claude down.

5
The layers are complementary, not alternatives

Intent without enforcement is a suggestion. Enforcement without intent is brittle. Signalling without enforcement is noise. All four together create a robust system.
