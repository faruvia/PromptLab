# create-doc Skill

## Introduction

`create-doc` is a Claude Code **project skill** — a reusable, slash-command-invocable unit of agent behaviour defined entirely in a single markdown file. When triggered, it accepts a free-text prompt describing what to document and writes a well-structured markdown file into the `docs/` folder.

It exists to enforce the project convention (declared in `CLAUDE.md`) that **all project markdown lives under `docs/`**, and to make that convention easily invocable without having to spell it out each time.

---

## Purpose

| Goal | How the skill addresses it |
|---|---|
| Consistent file placement | Always writes to `docs/`, never to the project root or source directories |
| Avoid duplicates | Checks for an existing file at the target path and updates it in place |
| Code-aware documentation | Reads relevant source files under `promptlab/` before writing technical docs |
| Repeatable quality | Enforces a fixed output format (headings, tables, fenced code blocks) |

---

## Architecture

Skills are discovered by Claude Code by scanning `.claude/skills/` at session startup. Each sub-folder becomes a command whose name equals the folder name.

```
.claude/
└── skills/
    └── create-doc/          ← folder name = command name
        └── SKILL.md         ← skill definition (frontmatter + instructions)
```

There is only one file — `SKILL.md`. No code, no runtime, no build step. The model reads the instructions in `SKILL.md` and executes them using its own reasoning and the tools available in the session.

---

## Structure of SKILL.md

```
.claude/skills/create-doc/SKILL.md
```

The file has two parts:

### 1. Frontmatter (YAML)

```yaml
---
name: create-doc          # must match the folder name exactly
description: "..."        # loaded into the model's context for auto-trigger detection
argument-hint: "..."      # shown in the slash-command picker UI
---
```

| Field | Purpose |
|---|---|
| `name` | Registers the command as `/create-doc` |
| `description` | Used by the model to decide when to auto-trigger the skill from natural language |
| `argument-hint` | Hint text shown in the IDE command picker |

### 2. Body (Markdown instructions)

The body contains step-by-step instructions the model follows when the skill runs. These are plain English — no code, no special syntax:

1. Derive a kebab-case filename from the user's argument
2. Check `docs/` for an existing file; update it rather than creating a duplicate
3. Read relevant source files from `promptlab/` if the subject is code-related
4. Write the finished document to `docs/<filename>.md`
5. Apply output formatting rules (headings, tables, fenced blocks; no "generated by" footer)

---

## How to Invoke the Skill

### Slash command
```
/create-doc <what to document>
```

Examples:
```
/create-doc API reference for all PromptLab endpoints
/create-doc architecture overview of the provider system
/create-doc getting started guide for new contributors
/create-doc explain the LLM service and how providers are selected
```

### Natural language (auto-triggered)
The model also invokes the skill automatically when it detects phrases like:

- "document X"
- "write docs for X"
- "create a guide for X"
- "add a markdown file about X"

---

## Filename Derivation Rules

The user's argument is converted to a filename by the model:

| Argument | Filename |
|---|---|
| `API reference` | `api-reference.md` |
| `architecture overview` | `architecture-overview.md` |
| `getting started guide for new contributors` | `getting-started-guide-for-new-contributors.md` |
| `explain create-doc skill` | `create-doc-skill.md` |

Algorithm: lowercase → spaces to hyphens → strip special characters → append `.md`.

---

## End-to-End Example

```
User:  /create-doc LLM service and how providers are selected

Model: 1. Derives filename → llm-service.md
       2. Checks docs/llm-service.md — does not exist, will create
       3. Reads promptlab/app/llm_service.py
       4. Reads promptlab/app/schemas.py (for ProviderConfig shape)
       5. Writes docs/llm-service.md with headings, a provider table,
          and a fenced code example
```

---

## Source Files the Skill Reads

When the documentation subject is code-related, the skill reads from:

| Path | Contents |
|---|---|
| `promptlab/app/routers/` | API route handlers |
| `promptlab/app/schemas.py` | Pydantic request/response models |
| `promptlab/app/crud.py` | Database access layer |
| `promptlab/app/llm_service.py` | LLM provider abstraction |
| `promptlab/app/models.py` | SQLAlchemy ORM models |

---

## Output Rules

- Clear headings (`##`, `###`)
- Tables for tabular data
- Fenced code blocks (with language tag) for examples
- No "generated by" footer
- File is always written to `docs/` — never to the project root or inside source directories

---

## Session Reload Requirement

Project skills are discovered at **session startup**. A skill created or modified during the current session requires a Claude Code restart (or VSCode extension reload) before it appears in the `/` command picker.

After reload, `/create-doc` is available as a registered slash command and is also auto-triggered by the model from natural language.
